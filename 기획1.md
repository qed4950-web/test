# FlavorOS – Recipe Strategy Engine

### (MD 기획서 최종본 + UI 흐름도 + 사업성 검증)

---

## 0. 한 줄 정의 (Pitch 문장)

> **FlavorOS는 ‘잘 팔리는 맛집의 성공 구조’를 분석해, 복제·근접·전환함으로써 새로운 히트 레시피를 설계하는 AI 레시피 전략 엔진다.**

---

## 1️⃣ MD 기획서 최종본 (Product Spec)

### 1.1 문제 정의 (Why Now)

* F&B 실패의 **90%는 운영 문제가 아니라 ‘메뉴 기획 단계’에서 이미 결정됨**
* 대부분의 브랜드는:

  * “감”으로 메뉴를 만든다
  * 경쟁 맛집을 **정량적으로 비교할 방법이 없다**
  * 성공한 맛을 **어디까지 따라가야 하는지 모른다**

👉 **FlavorOS는 ‘맛의 구조’를 좌표화**해 이 문제를 제거한다.

---

### 1.2 해결 방식 (What)

FlavorOS는 레시피를 **벡터(Vector)**로 표현하고,
두 레시피 사이의 **거리·방향·구조**를 계산한다.

* Reference1: **성공한 맛집 레시피**
* Reference2: **내 브랜드 레시피**
* 결과:

  * 얼마나 다른지
  * 어디가 다른지
  * 어떻게 바꾸면 되는지

---

### 1.3 핵심 기능 (Core Features)

#### (1) Recipe Reference System

* 맛집 레시피 / 내 레시피를 **동일한 좌표계**에 올림
* “비슷하다/다르다”가 아니라 **얼마나·어디서** 다른지 표시

#### (2) Flavor Vector Engine

* 맛을 다음 4계층으로 수치화

| Layer    | 구성                                   |
| -------- | ------------------------------------ |
| Taste    | Salt / Sweet / Sour / Bitter / Umami |
| Texture  | Fat / Crisp / Juiciness              |
| Aroma    | Fire / Garlic / Fermented / Spice    |
| Behavior | Addictiveness / Satiety / Repeat     |

> ❌ 건강, ❌ 원가율
> ✅ 오직 “잘 팔리는 맛의 구조”

---

#### (3) Strategy Actions (의사결정 엔진)

| 전략              | 의미                |
| --------------- | ----------------- |
| Copy            | reference1을 n% 복제 |
| Distance Reduce | 같은 방향, 브랜드 유지     |
| Direction Shift | 성공 축을 반대로 사용      |
| New Signature   | 성공 논리만 차용         |

각 선택은 **결과 예측**을 동반:

* 고객 반응
* 가격대 적합성
* 브랜드 충돌 위험

---

### 1.4 산출물 (Output)

* New Recipe Blueprint

  * 맛 벡터
  * 조리 포인트
  * 마케팅 문장 (POS/배너용)

다음 단계 제안 (진짜 다음 스텝):

* **① MVP 스코프 1페이지로 압축**
* **② 투자자용 10장 피치덱**
* **③ 실제 메뉴 하나를 reference1/reference2로 시뮬레이션**

👉 어디부터 갈지 번호로 말해.
이건 **이제 ‘아이디어’ 단계를 넘었다.**


->

좀더 ai를넣을순 없나

->

1️⃣ 지금 구조에서 AI가 이미 들어갈 수 있는 지점

지금 구조를 다시 보면:

Reference 수집
→ Flavor Vector
→ Distance / Direction
→ Strategy 선택
→ Recipe Blueprint


여기서 AI가 개입 가능한 레이어는 최소 5개다.

2️⃣ AI Level 1 — Reference 자동 생성 AI (지금 바로 가능)
🔹 뭐를 AI가 하냐

맛집 리뷰, SNS, 블로그 텍스트를 읽고

Reference1 Flavor Vector를 자동 추정

🔹 기존 방식

사람이 “이 집은 불향 쎄고 단짠” 판단

🔹 AI 방식

LLM + 키워드 가중치

예:

“불향 미쳤다” → Fire +0.8

“계속 생각남” → Addictiveness +0.7

📌 중요

이건 예측이 아니라 요약·추출

그래서 신뢰도 높음

👉 MVP에 가장 먼저 넣어야 할 AI

3️⃣ AI Level 2 — Distance 해석 AI (설명하는 AI)
🔹 기존

“Umami가 부족합니다” (숫자)

🔹 AI 추가

왜 이 차이가 문제인지 설명

예:

“reference1은 감칠맛을 지방과 불향으로 보강해 ‘먹고 난 뒤 기억’을 만드는데,
현재 레시피는 간장 베이스에 의존해 여운이 짧습니다.”

📌 포인트

AI는 결론을 내리지 않음

해석과 언어화만 담당

👉 사장/기획자가 AI를 신뢰하게 되는 지점

4️⃣ AI Level 3 — Strategy 추천 AI (의사결정 보조)

여기서부터 ‘AI 느낌’이 확 살아난다.

🔹 입력

Reference1 / Reference2

브랜드 톤

가격대

타겟 고객

🔹 AI 출력

“이 경우 Copy보다 Distance Reduce가 적합합니다.
이유: 브랜드 충돌 리스크 0.63, 가격대 미스매치 가능성”

📌 핵심

AI가 버튼을 누르지 않는다

추천 + 근거만 제시

👉 인간이 최종 결정 = 책임 구조 유지

5️⃣ AI Level 4 — 시뮬레이션 AI (가상 고객 반응)

여기서부터 투자자 설득용 AI가 된다.

🔹 기능

“이 레시피를 출시했을 때”

타겟 고객 반응을 시나리오로 생성

예:

직장인 남성:

“처음엔 센데, 계속 손이 감”

재구매 확률: 중상

가격 저항: 낮음

📌 이건 예측이 아니라
과거 패턴 + 언어 모델 기반 시뮬레이션

👉 메뉴 회의에서 압도적으로 강해짐

6️⃣ AI Level 5 — Recipe Mutation AI (고급, 후반)

이 단계가 되면 진짜 ‘AI 레시피 엔진’ 소리 듣는다.

🔹 개념

Reference1을 그대로 쓰지 않고

Flavor Vector를 변이(mutation)

예:

불향 유지

단맛 ↓

발효향 +0.3 추가

🔹 결과

“어디에도 없는 맛”

하지만 성공 구조는 유지

📌 이건 AI 단독 생성 ❌

반드시 인간이 선택한 전략 안에서만 동작

7️⃣ 정리: AI를 이렇게 포지셔닝해야 산다
❌ 나쁜 포지션

“AI가 레시피를 만들어줍니다”

“AI가 메뉴를 추천합니다”

→ 신뢰 없음, 책임 없음, 사업성 낮음

✅ 좋은 포지션

“AI가 성공한 맛의 구조를 해석하고,
인간의 판단을 가속·보조합니다”

AI = 분석가 + 시뮬레이터

인간 = 결정권자

8️⃣ FlavorOS의 AI 스택 한 줄 요약
Rule-based Flavor Engine (중심)
 + LLM (해석·추천·시뮬레이션)
 + 데이터 축적 (점점 정교)


👉 이 구조는:

과장되지 않고

투자자에게 설명 가능하고

MVP → 고도화 루트가 명확함
